Componentes Internos — Como Funciona Passo a Passo

Desde o frontend TUI até a detecção de agentes, cada componente tem um papel preciso na arquitetura do Herdr.

🖥️
Herdr Server (Background)
Processo daemon que gerencia todos os PTYs. Mantém estado de workspaces, tabs, panes e agentes. Persiste sessão em session.json. Não expõe portas de rede — comunica apenas via socket local.
Rust Binary • PID isolado
📺
Herdr Client (TUI)
Terminal User Interface que se conecta ao servidor. Renderiza panes, sidebar com estados de agentes, tabs. Suporta mouse nativo (click, drag, right-click). Múltiplos clientes simultâneos.
3 Modos: Terminal • Prefix • Navigate
🔌
Socket Local (IPC)
Unix Domain Socket (Linux/macOS) ou Named Pipe (Windows). Zero portas de rede. Protocolo NDJSON — cada mensagem é uma linha JSON terminada por \n. Request/Response com IDs.
~/.config/herdr/herdr.sock
⚡
PTY Manager
Gerencia pseudo-terminais reais. Cada pane possui seu PTY com shell/processo independente. Captura output, envia input, preserva scrollback. BSP layout tree para splits.
ioctl • TIOCGWINSZ • fork/exec
🔍
Agent Detection Engine
Detecta foreground process → avalia lifecycle hooks OU screen manifests (TOML) → classifica estado (idle/working/blocked/done/unknown). Bottom-buffer snapshot, não viewport scrollada.
Manifests: Bundled → Remote → Local Override
🤖
Agent Skill Layer
Agentes AI rodando dentro de panes (HERDR_ENV=1) podem usar CLI/Socket para orquestrar: criar panes, esperar output, ler resultados, spawnar outros agentes. É assim que agentes "veem" uns aos outros.
HERDR_ENV=1 • herdr CLI

Quem é o Orquestrador? — Fluxo de Orquestração

Resposta-chave: O Herdr não é um orquestrador top-down como Kubernetes. Ele é um observador inteligente + facilitador de comunicação. Cada agente AI mantém sua autonomia. O Herdr:

① Detecta agentes automaticamente (foreground process + manifests)
② Classifica estado semântico (blocked/working/done/idle)
③ Propaga estado via rollup (pane → tab → workspace → sidebar)
④ Facilita comunicação inter-agente via Socket API
⑤ Permite que agentes orquestrem uns aos outros via Agent Skill (herdr CLI)

Usuário inicia herdr
Server spawna (ou reattach). Cria workspace default com shell.

Usuário digita "claude" ou "codex" no terminal. Herdr injeta HERDR_ENV=1 e HERDR_SOCKET_PATH.

Server detecta foreground process, avalia screen manifests TOML, classifica estado como "idle".

Deep Dive — Sockets, Rede e Protocolo

Entenda como funciona a comunicação inter-processo, por que não há portas de rede, e como os agentes se comunicam.

Conceitos de Socket — Unix Domain Socket vs TCP

🔌 O que é um Unix Domain Socket?
Um Unix Domain Socket (UDS) é um endpoint de comunicação inter-processo (IPC) que existe como um arquivo no filesystem (ex: herdr.sock). Diferente de TCP sockets, ele não usa a stack de rede — a comunicação acontece inteiramente no kernel, sem overhead de protocolo de rede.

Herdr, tmux e GNU Screen todos usam Unix Domain Sockets. Nenhum abre portas TCP/UDP. A segurança é controlada por permissões do filesystem (chmod/chown).
📡 Herdr Socket API — Protocolo NDJSON
// Conectar ao socket socat - UNIX-CONNECT:~/.config/herdr/herdr.sock // Request (uma linha JSON + \n) {"id":"1","method":"ping","params":{}} // Response {"id":"1","result":{"type":"pong"}} // Listar agentes {"id":"2","method":"agent.list","params":{}} // Esperar estado de agente {"id":"3","method":"events.wait","params":{ "type":"pane.agent_status_changed", "pane_id":"w1:p1","agent_status":"done" }}
⚖️ UDS vs TCP — Comparativo

Escopo Local (máquina) Rede (qualquer IP)

🔄 Como Agentes se Comunicam
Agentes AI rodando dentro de panes Herdr não se comunicam diretamente entre si. Toda comunicação passa pelo Herdr Server como intermediário:

1. Agent A roda em pane w1:p1
2. Agent A usa herdr pane split → cria pane w1:p2
3. Agent A usa herdr pane run w1:p2 "codex" → spawna Agent B
4. Agent A usa herdr wait agent-status w1:p2 --status done
5. Agent A usa herdr pane read w1:p2 → lê resultado do Agent B

É um padrão de orquestração por observable state, não por mensagens diretas.

Gaps & Orquestração — O Que Falta para Projeto Real

Análise fundamentada na documentação oficial herdr.dev — O que o Herdr provê, o que falta, e soluções open-source para preencher cada gap.

⚡ Herdr = Infraestrutura, NÃO Orquestrador

✅ O que Herdr PROVÊ (Documentado)

  • ●PTY isolado por agente — herdr.dev/docs/concepts
  • ●Detecção automática de 17+ agentes — herdr.dev/docs/agents
  • ●5 estados semânticos com rollup — herdr.dev/docs/agents
  • ●Socket API NDJSON (~60 métodos) — herdr.dev/docs/socket-api
  • ●Git worktrees (worktree.create) — herdr.dev/docs/socket-api
  • ●Plugin system extensível — herdr.dev/docs/plugins
  • ●Safety rule HERDR_ENV=1 — herdr.dev/docs/agent-skill

❌ O que Herdr NÃO PROVÊ (Ausente na doc oficial)

  • ○Task management / Kanban board
  • ○Orquestração top-down (supervisor → workers)
  • ○File locking entre agentes
  • ○Check-ins / Status reports periódicos
  • ○Quality gates (test/lint before merge)
  • ○Contratos de escopo por agente
  • ○Task assignment automático

📋 Fonte: herdr.dev/docs/how-to-work — "Herdr is a background session server plus one or more terminal clients."

🧠 Quem é o Orquestrador?

Resposta fundamentada: Herdr não atribui tarefas a agentes. O fabricante define Herdr como background session server — não como "orchestrator" ou "task manager".

A orquestração acontece de 3 formas possíveis:
1. O humano decide o que cada agente faz (atribuição manual)
2. Um agente supervisor usa o Agent Skill para delegar via herdr pane split + herdr pane run
3. Tool externo (Taskter, CAO) gerencia o kanban e atribui tasks

📋 Fonte: herdr.dev/docs/agent-skill — "The skill is not a separate app or service. It is a markdown instruction file for agents."

Perguntas Críticas — Respostas Fundamentadas

📋 Check-ins?

❌ NÃO EXISTE NATIVAMENTE

Herdr provê event subscription (pane.agent_status_changed) e state rollup passivo, mas não check-ins programados.

Busca: "check-in", "standup", "report" — 0 resultados em herdr.dev

📝 Contratos?

⚠️ PARCIAL — Apenas Safety Rule

Existe 1 contrato: HERDR_ENV=1 — contrato de acesso. Não existe contrato de qualidade, escopo, ou SLA.

Fonte: herdr.dev/docs/agent-skill — "one guardrail"

🔒 Isolamento de Código?

⚠️ PARCIAL — PTY sim, Filesystem não

PTYs isolados ✅. Git worktrees disponíveis ✅. Mas file locking não existe — dois agentes podem editar o mesmo arquivo simultaneamente.
