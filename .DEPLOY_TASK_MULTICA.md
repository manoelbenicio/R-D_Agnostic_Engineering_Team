# DEPLOY TASK — Multica (Rotation-Parity Polyglot)

## PAPEL
Você é um agente de deploy. Sua única função é executar, na ordem, os passos de deploy definidos
abaixo — nada além, nada aquém. Você NÃO é um agente criativo: não refatora, não "melhora", não
adiciona nada que não esteja explicitamente escrito aqui.

## OBJETIVO
Fazer o deploy do sistema Multica (Rotation-Parity Polyglot) em ambiente Linux/WSL2 com Docker,
seguindo EXATAMENTE os PASSOS 1 a 5, validando cada etapa, e registrando cada tarefa em um arquivo de
check-in em disco. O escopo termina quando o CRITÉRIO DE SUCESSO for atingido.

## REGRAS INEGOCIÁVEIS
1. Faça SOMENTE o que está nos PASSOS. Nada a mais, nada a menos.
2. NÃO invente soluções, comandos, flags, caminhos, variáveis ou "correções" fora do escopo.
3. NÃO altere código-fonte, configs ou arquitetura. Este é um deploy, não um desenvolvimento.
4. NÃO copie arquivos .env de outra máquina. O passo 3 gera o .env automaticamente.
5. Em caso de QUALQUER dúvida, ambiguidade, erro não previsto ou desvio do resultado esperado:
   (a) PARE imediatamente; (b) registre a dúvida no check-in (status BLOCKED + descrição exata +
   comando/saída relevante, sem segredos); (c) pergunte ao responsável humano; (d) se NÃO houver
   resposta, PERMANEÇA PARADO. Jamais prossiga por conta própria nem improvise.
6. Nunca cole segredos (senhas, tokens, chaves) em logs, check-in ou respostas.

## PROTOCOLO DE CHECK-IN
- ANTES de iniciar, crie na RAIZ DO PROJETO o arquivo:
    ./CHECKIN_Codex55B_<UTC-ISO8601>.md
  (use o timestamp UTC atual)
- Para CADA passo/tarefa entregue, ANEXE uma entrada no formato:
    ## Codex55B · <UTC-ISO8601> · PASSO <n>
    status: DONE | BLOCKED
    comando: <o comando executado>
    resultado: <saída/evidência resumida e scrubbed>
    observacao: <opcional>
- Só marque um passo como DONE com evidência real (saída do comando). Sem evidência, não é DONE.
- Se bloquear, registre status: BLOCKED com o motivo exato e o que precisa para destravar.

## PRÉ-REQUISITOS
Verifique (se algum falhar, PARE e reporte conforme regra 5):
  docker --version && docker compose version
  git --version && make --version && openssl version
  Portas 8080 e 3000 livres. Se :3000 estiver ocupada, use FRONTEND_PORT=3100 no PASSO 3.

## PASSOS

### PASSO 1 — Atualizar o repositório
Você JÁ está no clone. Traga os últimos artefatos:
```bash
git fetch origin
git checkout main
git pull origin main
```
Esperado: "Already up to date" ou fast-forward até o último commit de origin/main; sem conflitos.

### PASSO 2 — Entrar na pasta da aplicação
```bash
cd multica-auth-work
```
Esperado: pasta contém Makefile e docker-compose.selfhost.yml.

### PASSO 3 — Subir a stack
```bash
make selfhost
```
(Se as imagens não puxarem: `make selfhost-build`. NÃO tente outra alternativa sem autorização.)
Esperado: 3 serviços no ar; .env gerado automaticamente; migrations aplicadas no log.

### PASSO 4 — Validar
```bash
docker compose -f docker-compose.selfhost.yml ps
curl -s http://localhost:8080/health
curl -s http://localhost:8080/readyz
curl -s -o /dev/null -w '%{http_code}\n' http://localhost:3000
```
Esperado: postgres (healthy); /health {"status":"ok"}; /readyz db:ok e migrations:ok; frontend 200.

### PASSO 5 — Primeiro login + CLI/daemon
Abra http://localhost:3000, informe um e-mail e obtenha o código:
```bash
docker compose -f docker-compose.selfhost.yml logs backend | grep -i "verification code"
```
Faça login e crie o primeiro workspace.
Instale a CLI multica e rode: `multica setup self-host`
Esperado: login bem-sucedido; workspace criado; daemon iniciado.
