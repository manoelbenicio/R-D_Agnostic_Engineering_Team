1 As 3 camadas

Tudo no sistema cai em uma de três camadas. O mesmo código roda em local e em cloud — o que muda é só onde ele é hospedado e como as credenciais chegam.

Bloqueio descoberto e testado: o cao-server é localhost-only por design. Rejeita Host de cloud (HTTP 400) e WebSocket não-loopback (403). Cloud exige patch no CAO (fork) ou proxy reverso reescrevendo Host→localhost. Ver §8.

6 O que é a "topologia"

Topologia = o grafo do canvas: os agentes (nós) e quem pode falar com quem (arestas). Você desenha isso no Canvas Builder. Tipos de aresta: handoff, assign, send_message.

Exemplo visual — e o que o "porteiro" (validation-proxy) faz

O VALIDATION-PROXY (faixa pontilhada no meio) é o porteiro: toda chamada entre agentes passa por ele antes de chegar ao destino. Setas sólidas = arestas que você desenhou → ele deixa passar. Seta tracejada vermelha = Developer→Reviewer sem aresta → ele trava e registra ali mesmo. Sem ele, o LLM poderia ignorar o prompt e fazer a chamada proibida.

Onde a topologia vive

Etapa Onde Arquivo
Você desenha tela do Canvas (navegador) src/canvas-builder/CanvasBuilderPage.tsx
Formato (o "molde") código TypeScript src/shared/canvas-types.ts → CanvasDocument.edges[]
Salvo IndexedDB (no navegador) src/canvas-document/store.ts
Aplicado (porteiro) compila edges → grafo src/shared/topology-guard.ts
Importante: sua topologia não é um arquivo no disco que você abre com 2 cliques. Ela mora dentro do navegador (IndexedDB) e você a vê/edita visualmente na tela do Canvas. O .ts é só a definição do formato (para desenvolvedores).

7 Fluxo de uma requisição (ponta a ponta)

  1. Você abre a SPA → React inicia, lê o IndexedDB.
  2. A SPA chama o runtime em VITE_CAO_BASE_URL (127.0.0.1:9889 local / URL do Cloud Run na nuvem).
  3. O runtime cria uma sessão + terminais (tmux), lançando um worker CLI (codex/kiro-cli/agy) por nó do canvas.
  4. O worker autentica usando as credenciais montadas (local) ou secrets (cloud).
  5. A saída do terminal volta pela SPA via WebSocket; a grade de terminais renderiza com xterm.
  6. Agentes coordenam via MCP (handoff/assign/send_message) — e é aqui que o validation-proxy checaria a topologia.
