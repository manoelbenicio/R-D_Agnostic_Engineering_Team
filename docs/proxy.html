O Proxy de Validação de Topologia

Onde fica, o que intercepta e como decide. É o "porteiro" das chamadas entre agentes.

1 Onde ele fica

O proxy não é um servidor separado e não fica entre o navegador e o runtime. Ele é um checkpoint dentro do cao-server, no plano de controle MCP — o único ponto por onde toda chamada agente→agente passa.

Repare: a chamada nunca vai direto de A para B/C. Ela sempre passa pelo proxy primeiro, porque é o cao-server (via MCP) quem entrega — e o proxy vive nesse ponto de entrega.

2 O fluxo, passo a passo

  1. Agente A (codex) decide delegar e chama a primitiva MCP handoff(target="Reviewer").
  2. A chamada sobe por MCP até o cao-server, que identifica o chamador pelo CAO_TERMINAL_ID.
  3. 🚦 O proxy intercepta aqui — antes da entrega. Resolve A → nó de origem, target → nó de destino, e consulta canvas.edges.
  4. Existe a aresta handoff: A → Reviewer no canvas? SIM → entrega o cao-server roteia para o Agente B.
  5. Não existe (ex.: A → agy)? NÃO → bloqueia retorna erro e grava um registro auditável; a chamada nunca chega ao destino.

3 Por que ele existe (o risco R3)

Sem o proxy, a única coisa que impede um agente de chamar outro fora do desenho é uma instrução no prompt ("seus alvos permitidos são X, Y"). Mas um LLM pode ignorar o prompt. Nada força o respeito à topologia.

CamadaO que fazGarantia
Prompt (canvas-topology)diz ao supervisor quais alvos são permitidosfraca — LLM pode ignorar
Validation-proxybloqueia a chamada se a aresta não existeforte — determinística
Analogia: o prompt é uma placa "entre só por esta porta". O proxy é a fechadura que realmente tranca as outras portas.
