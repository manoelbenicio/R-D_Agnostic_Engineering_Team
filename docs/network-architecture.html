1 Planta de rede (LOCAL)

Três zonas de confiança: a máquina host (seu WSL/Windows), o container Docker (sandbox isolado), e o mundo externo (APIs dos provedores). O navegador e o Docker rodam no host; o runtime e os agentes rodam dentro do container.

Host (sua máquina) Container Docker Externo (internet)
A fronteira de confiança mais importante: o navegador fala com o runtime só pelo loopback 127.0.0.1:9889. Os agentes (dentro do container) falam com as APIs externas em HTTPS :443 de saída, usando as credenciais montadas do host.
🚦 Onde entra o proxy: a faixa ciano no diagrama fica dentro do cao-server, no plano de controle MCP — exatamente na fronteira entre as sessões tmux (agentes) e o roteamento do servidor. Quando um agente (ex.: codex) chama handoff/assign/send_message para outro, a chamada sobe por MCP (seta vermelha) e o proxy a intercepta antes da entrega: aresta existe no canvas → entrega; não existe → bloqueia e registra. Note que o proxy não fica no caminho navegador↔runtime (loopback) nem na saída HTTPS — só no tráfego agente↔agente.

3 O "robô orquestrador" — onde está e o que faz

Quem é o orquestrador

É o cao-server (CLI Agent Orchestrator, da AWS Labs) — processo Python/uvicorn que roda dentro do container, na porta 9889. Ele não "pensa"; ele coordena:

  • cria sessões tmux (uma por agente);
  • lança o CLI worker certo em cada uma;
  • roteia mensagens entre agentes via MCP identificando cada um pelo CAO_TERMINAL_ID;
  • rastreia status (IDLE/PROCESSING/COMPLETED/ERROR);
  • faz stream do PTY de volta pela SPA via WebSocket.

Padrão: supervisor → workers

Um agente supervisor (entry point) delega para workers especialistas usando 3 primitivas MCP:

PrimitivaSemântica
handoffsíncrono — transfere e espera o resultado
assignassíncrono — dispara e segue (callback depois)
send_messagemensagem para a inbox de um agente existente
Essas 3 chamadas viajam por MCP dentro do cao-server — não passam pela SPA. É exatamente o caminho que o validation-proxy precisa interceptar.

O uvicorn escuta em 0.0.0.0:9889 dentro do container, mas o docker run -p 127.0.0.1:9889:9889 só expõe ao loopback do host. Além disso o próprio cao-server rejeita qualquer Host/IP não-loopback (defesa anti DNS-rebinding) — por isso o WebSocket fecha com código 4003 ip_not_allowed se a origem não for permitida.

Credenciaisbind-mount do hostSecret Manager → env
Rede do runtimeloopback-only ✓conflito rejeita host/IP não-loopback
Bloqueio de rede (testado): o cao-server foi desenhado para localhost-only. Em Cloud Run ele recebe um Host público (*.run.app) e conexões não-loopback → responde 400 no HTTP e fecha o WS com 4003 ip_not_allowed. Para cloud funcionar: (a) um reverse-proxy dentro do container reescrevendo Host→localhost e terminando o WS, ou (b) patch no CAO para liberar a origem do Cloud Run (= manter fork). Decisão pendente.
